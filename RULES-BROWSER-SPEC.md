# Magic Rules Browser Specification

## Purpose

Create a static web application for browsing the generated Magic: The Gathering Comprehensive Rules and glossary.

The application must support:

- hierarchical rules browsing
- glossary browsing
- deep linking to rules and glossary entries
- client-side full-text search
- tooltip previews for rules and glossary entries
- side-by-side cross-reference reading on larger screens

This specification reflects the decisions made during the design conversation and should be treated as the implementation baseline.

## Product Goals

- Provide a fast static reference browser for the Comprehensive Rules.
- Preserve the formal structure of rules and subrules.
- Make cross-references easy to inspect without losing reading context.
- Keep the deployment target simple: static files only.
- Keep the parser as the authoritative content transformation pipeline.

## Non-Goals

- No server-rendered application.
- No backend search service.
- No framework requirement such as React.
- No flattening of rules into prose-oriented content.
- No tooltip interaction model that turns previews into a second reading surface.

## High-Level Architecture

### Delivery Model

- The final product is a static site.
- The parser is the primary build system.
- `npm` dependencies are allowed when useful, but the site output must still be static.
- A separate frontend bundler is not the default architecture.

### Application Model

- The site is a single-page application.
- The SPA uses plain HTML, CSS, and JavaScript.
- The parser emits structured JSON bundles and prerendered HTML fragments used by the SPA.

### Output Strategy

- Keep the existing human-readable markdown tree as a parallel artifact.
- Also generate a site-oriented output tree under a single generated site root.
- The generated site root should contain everything needed to serve the application as static files.

## Content Sources

- Source of truth is the current official Comprehensive Rules TXT document discovered from the official rules page.
- The parser is responsible for checking whether the remote rules file is newer before downloading.
- Effective date metadata must be read from the source document, not hardcoded.

## Parser Responsibilities

The parser must:

- discover the current official Comprehensive Rules TXT URL from the official rules page
- compare remote metadata with local cached metadata
- download the TXT file only when the remote version is newer
- parse numbered rules into a structured hierarchy
- parse the glossary into a structured hierarchy
- preserve and render rule cross-links
- preserve and render glossary cross-links
- generate markdown output for human inspection
- generate SPA data bundles and prerendered HTML fragments
- generate search data
- generate tooltip preview data
- generate navigation data

## Site Output Layout

The exact directory names can vary, but the site output must be a single static root containing:

- `index.html`
- CSS assets
- JavaScript assets
- preload bundles for navigation, search, and tooltip previews
- lazily loaded content bundles for sections and glossary entries

The markdown output remains separate from the site output and continues to be generated.

## Rendering Format

- Rules and glossary content should be prerendered by the parser into HTML snippets.
- The reading pane must preserve rule structure, numbering, anchors, and subrule hierarchy.
- Tooltip previews should preserve inline formatting.
- Links inside tooltip content are interactive and can open nested previews.

## Navigation Model

### Global Navigation

- The application opens to an app-specific index page.
- The app-specific index page is not a raw mirror of the generated markdown index.
- It should act as the default browsing entry point for the SPA.

### Sidebar

- The left sidebar is collapsible.
- The sidebar has separate modes for `Rules` and `Glossary`.
- Rules and glossary must not be mixed into one combined tree.

### Rules Tree

- The rules tree includes all rules, including subrules.
- The tree hierarchy is generated by the parser, not derived in the browser.
- The rules tree is accordion-like:
  - only one chapter may be expanded at a time
  - within that chapter, only one section may be expanded at a time
  - subrules for the active target may be revealed automatically
- When a deep link is opened, the active path in the rules tree should auto-expand.

### Glossary Tree

- The glossary is browsed in its own sidebar mode.
- The glossary remains alphabetized and grouped.

### Mobile Navigation

- On small screens, the sidebar becomes an off-canvas drawer.
- The drawer should close after selection as appropriate for mobile use.

## URL and Routing Model

- Use hash-based routing.
- The URL encodes destination only.
- The URL must not encode transient sidebar state or live search text.
- If the app has no local sidebar state yet, sidebar mode may be inferred from the destination.
- Both rules and glossary entries must support deep links.

## Reading Pane Model

### Default Reading Behavior

- Navigating to a rule destination renders the full owning section page and scrolls to the target anchor.
- Navigating to a glossary destination renders the glossary entry page.

### Desktop and Larger Screens

- The application supports a primary reading pane and a secondary reading pane.
- The secondary pane exists to support cross-reference comparison.

### Mobile

- On mobile, the layout collapses to a single-pane reading model.

## Pane Navigation Rules

### Link Clicks in Reading Content

#### Same-Page Links

- If a clicked link targets a location within the same current page, the app scrolls to that location instead of opening a secondary pane.

#### Cross-Page Links from Primary Pane

- If a cross-page link is clicked in the primary pane:
  - open the target in the secondary pane to the right
  - if a secondary pane already exists, reuse it and navigate it to the new destination

#### Cross-Page Links from Secondary Pane

- If a cross-page link is clicked in the secondary pane:
  - promote the secondary pane to become the primary pane
  - open a new secondary pane with the clicked destination

### Non-Link Navigation

- Search results are not considered reading-pane navigation.
- Sidebar navigation and index-page navigation are separate from cross-reference pane navigation.
- If a non-link selection changes to a destination on a different page from the current primary page, reset to a single primary pane.

## Search Model

### Search Architecture

- Search is full-text.
- Search uses a ranked client-side index.
- Search requires no server.
- Use a small, modern, well-maintained library rather than a fully custom search engine.

### Search Results UX

- Search is a separate page, not an overlay inside the reading layout.
- The search page has no reading pane concept.
- Selecting a search result starts with a clean reading context.
- Search results are grouped by type rather than presented as one undifferentiated list.
- At minimum, results should be grouped into `Rules` and `Glossary`.
- Results should include a short matched snippet.
- Snippets are computed in the browser at query time.

### Search History

- Entering the search page must preserve the previous reading context in browser history.
- Going Back should restore the prior reading layout and scroll state.

## Tooltip Preview Model

### Supported Targets

- Tooltip previews work for both rule links and glossary links.

### Desktop and Keyboard

- Previews are available on hover.
- Previews are available on keyboard focus.

### Touch Devices

- Tapping a link opens a preview first.
- This applies to both same-page and cross-page targets.
- The preview should look visually consistent regardless of whether the target is same-page or cross-page.
- The preview includes an explicit link-to-full-page action using a hyperlink icon.

### Rule Preview Content

- A tooltip for a rule shows the targeted rule only.
- If the targeted rule has subrules, the preview should include a message indicating how many subrules exist.
- The preview should not automatically inline all direct child subrules.

### Glossary Preview Content

- Glossary previews should be smaller than full reading content but still formatted.

### Tooltip Interaction Constraints

- Tooltip content preserves inline formatting.
- Links inside tooltip content are interactive and can open nested previews.
- Glossary auto-linking should not be added inside tooltip previews.

## Scroll and History Behavior

- Each pane maintains its own scroll state while open.
- Navigating a pane to a new destination resets that pane to the new target anchor or the top of the new document.
- Browser history restoration must restore prior pane layout and prior scroll positions.
- Back and Forward navigation should restore the user’s prior reading context accurately.

## Glossary Auto-Linking Rules

Glossary auto-linking applies to the main reading pane content only.

### Matching Rules

- Matching is case-insensitive.
- Preserve the original displayed text.
- Prefer the longest matching glossary term.
- Exact full terms take priority over contained shorter terms.

Example:

- `Artifact Land` must be matched before `Artifact` or `Land`.

### First-Use Rules

- Auto-link only the first use of a canonical glossary term within a given section.
- Repeated uses of that same term in the same section remain plain text.
- Once a longer term claims an occurrence, shorter contained terms must not be linked inside that occurrence.
- If the longer term repeats later, do not fall back to linking shorter contained terms inside that repeated phrase.

### Variant Rules

- Support simple variants.
- Variants share the same first-use bucket as the canonical term.
- Variants should remain conservative and low-risk.

Supported low-risk normalization includes:

- singular and plural variants where the term boundary remains intact
- apostrophe normalization such as curly versus straight apostrophes
- other narrow normalization only when it is clearly deterministic

Do not introduce aggressive stemming or broad morphology inference.

## App Index Page

- The default application landing view is an app-specific index page.
- It should serve as a navigation-oriented start page for the SPA.
- It may include chapter and glossary entry points and lightweight guidance for search and browsing.
- It should not merely embed the raw generated markdown index unchanged.

## Accessibility and Interaction Expectations

- Hover previews alone are insufficient; focus support is required.
- Touch interaction must remain usable through preview-first behavior.
- The sidebar, search, and pane system should remain keyboard-usable.

## Implementation Defaults Locked In

The following are considered settled implementation defaults and should not be reopened without a new requirement:

- parser-driven static-site generation
- hash routing
- destination-only URLs
- prerendered HTML fragments generated by the parser
- grouped search results
- same-page destinations render in the current page context and scroll to anchor
- independent pane scroll state with restoration through history
- mobile collapses to single-pane reading and drawer navigation

## Suggested Output Bundles

The exact filenames are flexible, but the architecture should separate concerns into distinct bundles rather than one monolithic payload. A reasonable shape is:

- navigation bundle
- search index bundle
- tooltip preview bundle
- per-section rendered content bundles
- per-glossary-entry rendered content bundles

## Open Implementation Work

The following still need implementation, not product decisions:

- choose the exact search library
- define the concrete generated site directory layout
- implement parser-side JSON and HTML bundle generation
- implement glossary auto-linking with longest-match and first-use constraints
- implement the SPA shell, routing, search page, sidebar, and reading panes
- implement tooltip preview behavior across mouse, keyboard, and touch
- implement history restoration for pane layout and scroll state

## Acceptance Summary

The implementation is aligned with this specification when it provides:

- a static SPA generated by the parser
- a parser-generated rules hierarchy and glossary hierarchy
- prerendered reading content with preserved rule structure
- deep linking for rules and glossary entries
- dedicated search page with ranked full-text client-side search
- grouped results with snippets
- tooltip previews for rules and glossary entries
- dual-pane cross-reference reading on larger screens
- single-pane fallback on mobile
- conservative, first-use-only glossary auto-linking in the main reading pane
