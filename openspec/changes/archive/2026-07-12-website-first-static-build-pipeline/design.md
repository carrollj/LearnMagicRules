## Context

The current implementation generates a deploy-ready site root by having parser code copy website shell files and emit data into the same generated output tree. This creates inverted ownership: website UI source appears secondary while parser logic appears primary.

The target state is website-first. The website source becomes canonical application code, parser output becomes build input, and deployment remains static-only for GitHub Pages.

Constraints:
- GitHub Pages static hosting only (no backend/runtime server logic).
- Generated content should be CI-only and not committed.
- Public script contract should expose only `build` and `dev`.
- `dev` should depend on artifacts previously generated by `build`.

## Goals / Non-Goals

**Goals:**
- Establish website-first repository ownership and build flow.
- Integrate a bundler with SCSS support for frontend assets.
- Keep parser as a supplemental content generator that emits content bundles only.
- Ensure CI produces a fully static deploy artifact (`dist`) and deploys it to GitHub Pages.
- Keep developer workflow minimal and explicit: `build` then `dev`.

**Non-Goals:**
- Introducing any backend service, SSR, or runtime API.
- Expanding public scripts beyond `build` and `dev` in this change.
- Redesigning parser semantic parsing behavior unrelated to output ownership and pipeline integration.

## Decisions

### 1. Website-first ownership boundary
- Decision: Website app source becomes canonical; parser code is treated as a tooling pipeline.
- Rationale: Matches product reality and removes confusion about where UI behavior should be changed.
- Alternative considered: Keep current parser-owned site shell copy model.
- Why rejected: Preserves the current inversion and couples parser internals to website shell lifecycle.

### 2. Parser emits content bundles only
- Decision: Parser no longer copies `index.html` or static UI assets. It writes content bundles to a build workspace consumed by the web build.
- Rationale: Creates a clear contract between content generation and app presentation.
- Alternative considered: Continue emitting complete site output from parser.
- Why rejected: Blends concerns and makes frontend evolution harder.

### 3. Bundled frontend with SCSS, static output
- Decision: Use a bundler pipeline that compiles SCSS and outputs static files into `dist`.
- Rationale: Enables maintainable asset pipeline while preserving static hosting compatibility.
- Alternative considered: Keep plain copied assets without bundling.
- Why rejected: Conflicts with planned SCSS adoption and limits future frontend maintainability.

### 4. CI-only generated content
- Decision: Generated content artifacts are created during CI/build and are not committed.
- Rationale: Prevents stale generated files, reduces repository noise, and keeps source-of-truth clean.
- Alternative considered: Commit generated bundles.
- Why rejected: Increases churn and merge conflicts for deterministic machine output.

### 5. Minimal public script contract
- Decision: Expose only `build` and `dev`.
- Rationale: Avoids speculative commands and keeps user-facing workflows simple.
- Alternative considered: Add `parse`, `build:web`, and refresh scripts now.
- Why rejected: Not justified by current workflow needs.

## Risks / Trade-offs

- [Build runtime increases because `build` always includes full parse] -> Mitigation: keep parser deterministic and optimize independently if needed later.
- [Developers may run `dev` before `build` and encounter missing artifacts] -> Mitigation: fail fast with actionable guidance to run `build` first.
- [Migration touches parser paths, frontend paths, and CI together] -> Mitigation: stage migration with contract checks and explicit acceptance criteria.
- [Bundler migration may introduce subtle path/base issues for GitHub Pages] -> Mitigation: validate hash routing and asset base paths in CI preview before rollout.

## Migration Plan

1. Establish website source location and bundler config with SCSS support.
2. Update parser output contract to emit content bundles only into a build workspace.
3. Rewire website data loading to consume generated bundles from build output.
4. Update scripts to the minimal contract (`build`, `dev`) and implement `dev` precondition checks.
5. Update GitHub Actions to run `build` and deploy `dist`.
6. Remove legacy parser-owned shell copy behavior and obsolete output assumptions.

Rollback strategy:
- Revert workflow and script changes together to restore previous parser-generated deploy root behavior.
